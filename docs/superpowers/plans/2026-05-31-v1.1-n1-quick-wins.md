## N1-T1 — Recover pending apply-review on WS reconnect (frontend)

### Problem
`GET /api/applications/{job_id}/review-state` is implemented + backend-tested
(`tests/test_review_state.py`) but **no frontend code calls it**. If a client loses its WebSocket
while an auto-apply is paused awaiting confirmation (the `apply_review` modal), reconnecting (or
reloading the page) silently loses the in-flight review — the user can neither confirm nor cancel.

### Key facts (verified)
- The review modal lives in `frontend/src/routes/queue/+page.svelte` as `confirmModal`
  (`{ jobId, method, fields?, screenshot? }`). It is populated by a `$effect` watching `$messages`
  for `apply_review` messages (`{ type, job_id, filled_fields, screenshot_base64 }`).
- The WS store exposes `onWsConnect(cb)` (`frontend/src/lib/stores/websocket.ts`) which fires the
  callback on every successful (re)connection and returns an unsubscribe fn. The queue page already
  uses it for `syncBatchStatus`.
- The HTTP endpoint returns the **engine snapshot shape** (`backend/applier/engine.py`
  `get_pending_review`): `{ "job_id": int, "filled_fields": dict, "screenshot_b64": str | null }`.
  ⚠️ Note the key is `screenshot_b64` here, vs `screenshot_base64` on the WS `apply_review` message.
- `apiFetch<T>(path)` (`frontend/src/lib/api.ts`) throws on non-2xx; a 404 (no pending review) must be
  treated as "nothing to recover", not an error to surface.

### Approach (TDD)
1. **New pure module** `frontend/src/lib/utils/pendingReview.ts`:
   - `STORAGE_KEY = 'jobpilot:pending-reviews'`.
   - `loadPendingReviewIds(): number[]` — read + JSON-parse the sessionStorage set, tolerate missing/
     corrupt values (return `[]`). Guard `typeof window`/`sessionStorage` for SSR safety.
   - `savePendingReviewIds(ids: number[]): void` — persist (dedup, numbers only).
   - `addPendingReviewId(id)` / `removePendingReviewId(id)` — convenience read-modify-write helpers
     returning the new array.
   - `reviewStateToModal(payload)`: maps the **HTTP** snapshot
     `{ job_id, filled_fields, screenshot_b64 }` → `{ jobId, method: 'auto', fields, screenshot }`
     (i.e. `screenshot_b64` → `screenshot`, `filled_fields` → `fields`). Return `null` for a
     null/!object payload.
2. **Tests** `frontend/src/lib/utils/pendingReview.test.ts` (vitest, mock `sessionStorage` via a
   simple in-memory stub on `globalThis`): round-trip load/save, corrupt JSON → `[]`, add/remove
   dedup, `reviewStateToModal` maps keys correctly incl. null screenshot, null payload → null.
3. **Wire into** `frontend/src/routes/queue/+page.svelte`:
   - When `confirmModal` is set from an `apply_review` message, also
     `savePendingReviewIds(addPendingReviewId(loadPendingReviewIds(), jobId))`.
   - In `confirmApply()` and `cancelApply()`, remove that jobId from the persisted set.
   - Add a `recoverPendingReviews()` async fn: for each id in `loadPendingReviewIds()`, if no
     `confirmModal` is currently shown, `apiFetch` the review-state endpoint inside try/catch; on
     success map via `reviewStateToModal` and set `confirmModal`; on 404/any error, drop that id from
     the persisted set (the review is gone). Stop after the first one that yields a modal (single-modal UI).
   - Register it on connect alongside the existing one: keep `syncBatchStatus` and add
     `recoverPendingReviews` — either via a second `onWsConnect(...)` (store both unsub fns and call
     both in `onDestroy`) or a combined callback. Also call `recoverPendingReviews()` once in
     `onMount` (covers the reload-while-paused case, since the WS may already be open).
4. Do NOT change the `apply_review` WS path or the backend.

### Acceptance
- vitest: new tests pass; existing vitest suite unaffected.
- `svelte-check`: 0 errors / 0 warnings.
- Manually reasoned flow: apply_review → reload page → on (re)connect the modal reappears with the
  same fields/screenshot; confirm/cancel clears the persisted id so it doesn't re-appear.

### Files
- New: `frontend/src/lib/utils/pendingReview.ts`, `frontend/src/lib/utils/pendingReview.test.ts`
- Modified: `frontend/src/routes/queue/+page.svelte`

## N1-T2 — Align `batch_runner` ws-stub signatures (pyright)

### Problem
`backend/scheduler/batch_runner.py:35-44` defines `except`-fallback stubs for when
`backend.api.ws` can't be imported. Their parameter names are underscore-prefixed
(`_message`, `_progress`, `_match_id`, `_ats_score`, `_gap_severity`, `_decision`, `_covered`,
`_gaps`) while the real functions are:
- `broadcast_status(message: str, progress: float = 0.0)`
- `broadcast_job_assessment(match_id, ats_score, gap_severity, decision, covered, gaps)`
Callers pass these by keyword, so pyright reports ~9 "no parameter named ..." errors against the
fallback signatures.

### Approach
- Rename the fallback stub parameters to **exactly** match the real `ws.py` signatures (drop the
  underscore prefixes); keep type hints aligned (`message: str`, `progress: float = 0.0`,
  `match_id: int`, `ats_score: float`, `gap_severity: float`, `decision: str`, `covered: list`,
  `gaps: list`). Bodies stay `pass`. Keep the `# type: ignore[misc]` comments only if still needed.
- Ruff selects only `E`/`F`/`I` (see `pyproject.toml`), so unused-argument lint won't fire on the
  stubs; no `del`/`noqa` gymnastics needed.

### Acceptance
- `uv run pyright backend 2>&1 | tail -1` shows the count dropped by the stub errors (target: the ~9
  `batch_runner` ws-stub errors gone). Record before/after counts in the report.
- No behavior change; `uv run pytest -q` stays green.

### Files
- Modified: `backend/scheduler/batch_runner.py`

### Commit
`fix(N1): align batch_runner ws-stub signatures (pyright)`
