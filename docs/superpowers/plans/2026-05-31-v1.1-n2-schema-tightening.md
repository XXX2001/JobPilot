## Shared context (read once, applies to every N2 task)

### How the schema is built & tested (CRITICAL)
- `tests/conftest.py` bootstraps each test DB via `backend.database.init_db()`, which runs
  **`alembic upgrade head`** — NOT `Base.metadata.create_all`. So **a constraint only exists in tests
  if it's in a migration.**
- `tests/test_migrations.py::test_upgrade_head_is_clean_and_in_sync` upgrades a fresh temp DB to head
  and asserts `compare_metadata(mc, Base.metadata) == []`. **This is the canary**: every task must
  leave this green, which forces the model and the migration to agree.
  - `compare_metadata` DOES detect: `nullable` mismatches, missing/extra `UniqueConstraint`s, missing/
    extra `Index`es. → those MUST be added to BOTH the model and the migration.
  - `compare_metadata` does NOT reflect `CheckConstraint`s on SQLite → adding a CHECK won't break the
    canary either way, but you MUST still add it to BOTH the model `__table_args__` (documentation +
    consistency) AND the migration (so it's actually enforced in the test DB).
- Constraint-rejection tests follow `tests/test_db_integrity.py`: raw `INSERT … VALUES` via a session
  and `pytest.raises(IntegrityError)`. These run against the bootstrapped (upgraded) DB.

### SQLite migration pattern (follow T2a exactly)
- Current Alembic head: **`e5a65a3427cf`** (`alembic/versions/e5a65a3427cf_t2a_schema_catchup_and_fks.py`).
- SQLite can't `ALTER TABLE ADD CONSTRAINT` / `ALTER COLUMN`. For NOT NULL / CHECK / UNIQUE on existing
  columns use `with op.batch_alter_table("<t>", recreate="always") as batch:` then
  `batch.create_check_constraint(name, "<sql>")`, `batch.create_unique_constraint(name, [cols])`, or
  `batch.alter_column(col, nullable=False)`. Additive indexes use `op.create_index(...)` ([safe], no recreate).
- **Name every constraint explicitly** (`ck_<table>_<col>`, `uq_<table>_<cols>`, `ix_<table>_<cols>`)
  so `downgrade()` can drop it and batch mode can manage it.
- **Data-clean BEFORE constraining** with `bind.execute(sa.text("..."))`, mirroring T2a's upgrade()
  step 1. A fresh test DB is empty (cleanup is a no-op there) — the cleanup protects real user DBs.

### One chained revision per task
Each task creates its OWN Alembic revision chained on the previous task's (T1→T2→T3→T4), so each task
is independently reviewable and leaves the canary green:
`e5a65a3427cf` → `t2b1_*` (T1) → `t2b2_*` (T2) → `t2b3_*` (T3) → `t2b4_*` (T4).
Generate with `uv run alembic revision -m "..."` (gives a hash id) and hand-write upgrade/downgrade,
OR hand-author the file with an explicit `revision`/`down_revision` like the T2a file. Verify the chain
with `uv run alembic heads` (must show exactly ONE head) after each task.

### Verification gate for EVERY N2 task
From repo root:
- `uv run pytest -q` — green (incl. your new constraint tests + the existing `test_migrations.py` /
  `test_db_integrity.py`).
- `uv run alembic upgrade head` on a throwaway DB AND `uv run alembic heads` shows a single head.
- `uv run pyright backend 2>/dev/null | grep -E "errors?,"` — at/under 19 errors / 8 warnings.
Record exact counts in the report.

## N2-T1 — Enum CHECK constraints + legacy status migration

**Revision:** `t2b1_enum_checks` (down_revision `e5a65a3427cf`).

### Scope
Add named `CheckConstraint`s for the enum-like string columns, after normalising any out-of-vocabulary
existing rows. Do the **legacy `Application.status` migration first** (`manual`/`assisted` → `applied`).

Columns to constrain (model `__table_args__` + migration, both):
- `applications.status` → `ck_applications_status` IN the `APPLICATION_STATUSES` set.
- `applications.method` → `ck_applications_method` IN `{auto, assisted, manual}`.
- `job_matches.status` → `ck_job_matches_status` IN the confirmed JobMatch set (incl. `selected`).
- `tailored_documents.doc_type` → `ck_tailored_documents_doc_type` IN `{cv, letter}`.
- `application_correspondence.direction` → `ck_application_correspondence_direction` IN `{inbound, outbound}`.
- `gmail_messages.category` → `ck_gmail_messages_category` IN the heuristic set, **but allow NULL**
  (column is nullable; write the SQL as `category IS NULL OR category IN (...)`).
- `search_settings.cv_modification_sensitivity` → `ck_search_settings_cv_sensitivity` IN
  `{conservative, balanced, aggressive}`.

For each nullable column being checked, the CHECK must permit NULL (`col IS NULL OR col IN (...)`).

### Data-cleanup (migration upgrade(), before adding the CHECKs)
- `UPDATE applications SET status='applied' WHERE status IN ('manual','assisted')`.
- For every other checked column, normalise stragglers: run a guarded
  `UPDATE … SET <col>=<fallback> WHERE <col> IS NOT NULL AND <col> NOT IN (...)` — pick a safe fallback
  per column (e.g. `gmail_messages.category` → `'unknown'`; `job_matches.status` → `'new'`). Document each.

### TDD
1. Tests first in a new `tests/test_schema_checks.py`:
   - For each constrained column: a raw INSERT with an out-of-vocab value raises `IntegrityError`;
     a valid value (and NULL where allowed) inserts fine. Mirror `test_db_integrity.py` style (build
     the minimal parent rows needed for FKs, or insert into tables without FK deps).
   - A legacy-migration test: with a temp DB, `command.upgrade(cfg, "e5a65a3427cf")`, INSERT an
     `applications` row with `status='manual'`, `command.upgrade(cfg, "t2b1_enum_checks")` (or `head`),
     then assert the row's status is now `applied`. (Reuse the `_alembic_config` helper pattern from
     `tests/test_migrations.py`.)
2. Implement model `__table_args__` CHECKs + the migration. Keep `test_migrations.py` green.

### Files
- Modified: `backend/models/application.py`, `backend/models/job.py`, `backend/models/gmail.py`,
  `backend/models/user.py` (for `cv_modification_sensitivity` on `SearchSettings`).
- New: `alembic/versions/t2b1_enum_checks_*.py`, `tests/test_schema_checks.py`.

### Commit
`feat(N2): CHECK constraints on enum-like columns + legacy status migration`

## N2-T2 — Unique constraints on natural keys

**Revision:** `t2b2_unique_keys` (down_revision = T1's revision).

### Scope
- `job_sources.name` → UNIQUE (`uq_job_sources_name`). It's currently `index=True` only
  (`backend/models/job.py:28`). Make the model `unique=True` and migrate. Dedup duplicates first.
- `(job_matches.job_id, job_matches.batch_date)` → UNIQUE. Today there's a **non-unique** index
  `ix_job_matches_job_id_batch_date` (`backend/models/job.py:70`). Replace it with a UNIQUE constraint
  (or unique index) named `uq_job_matches_job_id_batch_date`. Update the model `__table_args__` so the
  canary stays green (remove the now-redundant non-unique Index for that pair, or convert it). Dedup
  existing `(job_id, batch_date)` collisions first (keep the highest-id / latest row, delete the rest —
  document the rule). NOTE `batch_date` is nullable; SQLite treats multiple NULLs as distinct, so the
  constraint only bites for non-NULL batch_date rows — that's the intended dedup target.

### Data-cleanup (migration upgrade(), before the constraints)
- `job_sources`: collapse duplicate names — keep one row per name, repoint `jobs.source_id` of the
  losers to the survivor, then delete losers. (Write explicit SQL; document the survivor rule, e.g.
  smallest id.)
- `job_matches`: delete duplicate `(job_id, batch_date)` rows where `batch_date IS NOT NULL`, keeping
  the max(id) per group; first repoint/cascade any children (`tailored_documents`, `applications`) —
  but since `tailored_documents.job_match_id` is CASCADE and `applications.job_match_id` is SET NULL,
  deleting a dup match will cascade/null appropriately; prefer keeping the row that HAS children.
  Keep it simple and correct; document the rule.

### Optional follow-up (within this task, only if low-risk)
Simplify the read-then-write dedup in `backend/scheduler/batch_runner.py` `_store_matches` to lean on
the new unique constraint (e.g. catch IntegrityError / use INSERT OR IGNORE). ONLY do this if it stays
correct and tested; otherwise leave the code path as-is and just note it. Don't break existing
`tests/test_batch_runner.py`.

### TDD
1. Tests first (extend `tests/test_schema_checks.py` or a new `tests/test_schema_unique.py`):
   - Duplicate `job_sources.name` insert raises `IntegrityError`.
   - Duplicate `(job_id, batch_date)` with non-NULL date raises; two rows with NULL batch_date for the
     same job do NOT raise (SQLite NULL semantics).
   - A migration test: build a pre-T2 DB, insert duplicates, upgrade, assert dedup happened and the
     constraint now holds.
2. Implement. Keep `test_migrations.py` + `test_batch_runner.py` green.

### Files
- Modified: `backend/models/job.py` (+ maybe `backend/scheduler/batch_runner.py`).
- New: `alembic/versions/t2b2_unique_keys_*.py`, tests.

### Commit
`feat(N2): unique constraints on job_sources.name and (job_id, batch_date)`

## N2-T3 — NOT NULL / conditional CHECK on always-set columns

**Revision:** `t2b3_not_null` (down_revision = T2's revision).

### Scope
- `tailored_documents.job_match_id` → NOT NULL (always written with a match:
  `backend/scheduler/batch_runner.py`, `backend/api/documents.py`). Delete orphans
  (`job_match_id IS NULL`) first. Update model (`nullable=False`) + migration `alter_column`.
- `jobs.dedup_hash` → NOT NULL (dedup invariant; currently `nullable=True, unique=True`,
  `backend/models/job.py:62`). Backfill or delete rows with NULL hash first. Keep `unique=True`.
- `applications.job_match_id`: the manual-apply API path **legitimately** allows NULL
  (`POST /api/applications`). So do NOT blunt-NOT-NULL it. Instead add a conditional CHECK
  `ck_applications_job_match_required` = `method = 'manual' OR job_match_id IS NOT NULL`. CONFIRM the
  manual path actually inserts NULL job_match_id before choosing this; if in doubt, report and ask.
- Explicit `nullable=False` where a column is de-facto required but currently relies only on a Python
  default and it's [safe]: `job_matches.job_id` (FK, already effectively required), `job_matches.status`,
  `application_events.application_id`. Set these in the model AND the migration `alter_column` so the
  canary agrees. (These should already be non-null in practice; the migration recreate makes it real.)

### Data-cleanup (upgrade(), before constraints)
- `DELETE FROM tailored_documents WHERE job_match_id IS NULL`.
- `jobs.dedup_hash`: backfill a deterministic hash where NULL if feasible, else
  `DELETE FROM jobs WHERE dedup_hash IS NULL` (document the choice; deletion cascades to job_matches).
- For the conditional `applications` CHECK: normalise any row that violates it (e.g. a non-manual row
  with NULL job_match_id) — decide delete vs set `method='manual'`; document.

### TDD
1. Tests first: NULL insert into each now-NOT-NULL column raises `IntegrityError`; a manual-method
   `applications` row with NULL `job_match_id` still inserts; a non-manual row with NULL job_match_id
   raises. Migration test: pre-state with NULLs → upgrade cleans → constraint holds.
2. Implement. Keep all existing tests green (watch for tests/fixtures that insert rows omitting these
   columns — fix them to supply values, or confirm they always did).

### Files
- Modified: `backend/models/document.py`, `backend/models/job.py`, `backend/models/application.py`.
- New: `alembic/versions/t2b3_not_null_*.py`, tests.

### Commit
`feat(N2): NOT NULL + conditional CHECK on always-set columns`

## Explicitly NOT in N2 (deferred — see spec §5)
- `String(n)` length caps (data-truncation risk).
- `server_default`/`onupdate` on timestamp columns (recreate cost; Python defaults already work).
- Singleton `CHECK(id=1)` + boot-seeding for `UserProfile`/`SearchSettings`.
