## 1. Goal & scope

The v1 roadmap (M1–M4) shipped a complete, documented, installable product. This follow-up
closes the **remaining non-deferred work** identified by the v1 final review and the standing
backlog, in three milestones:

- **N1 — Quick wins:** the two non-blocking follow-ups from the v1 final review.
- **N2 — T2b schema tightening:** the last untouched follow-on track (DB constraint hardening).
- **N3 — Packaging/CI real validation:** prove the install + CI end-to-end (the one thing v1
  could not verify in-sandbox).

**Explicitly out of scope** (deferred by design in the v1 spec §3, unchanged here): Calendar/CalDAV,
multi-user/auth/billing, mobile app, the deep apply-flow rewrites (T4b's `ApplicationEngine`
decomposition R4 + `BatchRunner`→`Phase` R2), and new job-board integrations.

## 2. Acceptance bar

1. The `review-state` endpoint is consumed end-to-end (a client that reconnects mid-apply
   recovers the pending review), or the unused endpoint is removed — no phantom surface.
2. The `batch_runner` fallback-stub pyright errors are gone (pyright ≤ 18 errors / 8 warnings
   after this work, down from 27 — the ~9 stub errors removed; the CI ceiling is ratcheted to match).
3. T2b: enum-like columns carry DB-level CHECK constraints, the documented natural keys are
   UNIQUE, the always-set FKs are NOT NULL, and the hot query paths have covering indexes —
   all via an idempotent Alembic migration that data-cleans before constraining, proven by tests.
4. One real Docker image build + `docker compose up` smoke succeeds (documented), CI runs green
   on the remote at least once, README badge points at the real repo.
5. No regressions: backend suite green, `svelte-check` 0/0, `vitest` green, `pyright` at/under the
   ratcheted ceiling.

## 3. Working method

- Branch `feat/v1.1-finishing-touches` (off `main`), merged locally when done. No push without the
  user's go-ahead (N3 explicitly involves pushing — gated on approval).
- Subagent-driven development, TDD, per-task spec + quality review, per-milestone final review.

## 4. Milestone N1 — Quick wins (from the v1 final review)

### N1-T1 — Wire `review-state` re-fetch on the frontend (close the phantom-surface gap)
`GET /api/applications/{job_id}/review-state` is functional + tested but never called by the UI,
so a client that drops its WebSocket mid-apply never recovers the pending `apply_review` payload.

- Track in-flight apply job ids on the queue page (`frontend/src/routes/queue/+page.svelte`):
  when an apply is initiated and while `confirmModal` is active, record the `job_id` in a small
  `$state` set, persisted to `sessionStorage` so it survives a page reload.
- On WS (re)connect — reuse the existing `onWsConnect(...)` hook (already wired to
  `syncBatchStatus`) — for each tracked in-flight `job_id`, `apiFetch` the `review-state` endpoint;
  if it returns a payload (filled_fields + screenshot), repopulate `confirmModal` so the review
  modal reappears. Clear the tracked id on confirm/cancel/terminal.
- Extract the "which job ids to re-check" / payload→modal mapping into a small pure function with a
  `vitest` test.
- If, on closer inspection, tracking in-flight ids proves disproportionate, the fallback is to
  REMOVE the endpoint + its test instead (honest-surface) — but the wiring is preferred since the
  backend already pays for it.

**Tests:** vitest for the pure mapping/selection logic; `svelte-check` 0/0. Backend already covered
by `tests/test_review_state.py`.
**Commit:** `feat(N1): recover pending apply-review on WS reconnect`.

### N1-T2 — Fix `batch_runner` fallback-stub pyright errors
`backend/scheduler/batch_runner.py:39-44` defines `except`-fallback stubs whose params are
`_message`/`_progress`/`_match_id`/`_ats_score`/… while the real `ws.py` functions are
`broadcast_status(message, progress=0.0)` and
`broadcast_job_assessment(match_id, ats_score, gap_severity, decision, covered, gaps)`. Callers use
keyword args, so pyright flags ~9 errors against the mismatched stub signatures.

- Rename the stub parameters to EXACTLY match the real function signatures (`message`, `progress`,
  `match_id`, `ats_score`, `gap_severity`, `decision`, `covered`, `gaps`). Bodies stay `pass`.
  (Ruff config selects only E/F/I, so unused-arg lint won't fire.)
- Confirm the ~9 pyright errors disappear; note the new count.

**Tests:** none needed (no behavior change); suite stays green.
**Commit:** `fix(N1): align batch_runner ws-stub signatures (pyright)`.

## 5. Milestone N2 — T2b schema tightening

Harden the DB schema with the constraints T2a left out. SQLite has limited `ALTER`, so all
constraint additions on existing columns use `op.batch_alter_table(..., recreate="always")` (the
T2a pattern in `alembic/versions/e5a65a3427cf_...py`), and every constraint is preceded by a
data-cleanup/audit step. One new Alembic revision `t2b_schema_tightening` chained on the current
head `e5a65a3427cf`. Each lot is a task: model change + migration step + tests (constraint rejects
bad rows; valid rows pass; `compare_metadata == []` after upgrade).

Reference vocabularies (for CHECKs) — from the live code:
- `Application.status` → `backend/applier/__init__.py` `APPLICATION_STATUSES` (`pending, applied,
  cancelled, failed, interview, offer, rejected`); legacy `manual`/`assisted` rows MUST be migrated
  first via `scripts/migrate_legacy_applied.py`.
- `Application.method` → `auto, assisted, manual`.
- `JobMatch.status` → `new, skipped, applying, applied, rejected, selected` (union of `queue.py` +
  `today.py` — VERIFY by `SELECT DISTINCT status` before constraining).
- `TailoredDocument.doc_type` → `cv, letter`.
- `ApplicationCorrespondence.direction` → `inbound, outbound`.
- `GmailMessage.category` → `noise, rejection, offer, interview_invite, ats_ack, unknown`
  (`classifier_heuristics.py`) — confirm the live distinct set, include any LLM-phase values.
- `SearchSettings.cv_modification_sensitivity` → `conservative, balanced, aggressive`.

### N2-T1 — Enum CHECK constraints + legacy data migration
Add `CheckConstraint`s for the enum-like columns above. Run the legacy-status migration
(`manual`/`assisted` → `applied`) and a distinct-value audit/normalisation for each column FIRST so
the constraints don't reject existing rows. Each CHECK must include exactly the live vocabulary
(don't over-restrict; e.g. include `selected` for `JobMatch.status`).
**Tests:** inserting an out-of-vocabulary value raises `IntegrityError`; every value the app actually
writes is accepted; migration is idempotent.
**Commit:** `feat(N2): CHECK constraints on enum-like columns + legacy migration`.

### N2-T2 — Unique constraints on natural keys
- `JobSource.name` UNIQUE (lookup key in `settings.py`) — merge duplicates first.
- `(JobMatch.job_id, JobMatch.batch_date)` UNIQUE (replaces the read-then-write dedup in
  `batch_runner.py`) — dedupe existing rows first. After the constraint lands, simplify the dedup
  code path to rely on the DB (optional within this task; keep behavior correct).
**Tests:** duplicate insert raises; the dedup path still works; migration idempotent + dedupes.
**Commit:** `feat(N2): unique constraints on job_sources.name and (job_id, batch_date)`.

### N2-T3 — NOT NULL on always-set columns
- `TailoredDocument.job_match_id` NOT NULL (always written with a match) — delete/backfill orphans first.
- `jobs.dedup_hash` NOT NULL (dedup invariant) — backfill/delete hash-less rows first.
- `Application.job_match_id`: the manual-apply API path legitimately allows `NULL`, so prefer a
  conditional `CheckConstraint("method = 'manual' OR job_match_id IS NOT NULL")` rather than a blunt
  NOT NULL. Confirm the manual path before choosing.
- Add explicit `nullable=False` to columns that are de-facto required but only Python-defaulted
  (`JobMatch.job_id`, `JobMatch.status`, `ApplicationEvent.application_id`, audit columns) where it's
  [safe] (no recreate-truncation risk).
**Tests:** NULL insert into a now-NOT-NULL column raises; the manual-apply path still inserts; migration
idempotent + cleans orphans.
**Commit:** `feat(N2): NOT NULL / conditional CHECK on always-set columns`.

### N2-T4 — Covering indexes for hot query paths
- `job_matches (status, batch_date, score)` for the queue listing (`queue.py`).
- `tailored_documents (job_match_id, doc_type, created_at)` for the documents lookups.
- `applications (status, created_at)` for the tracker listing.
Additive `create_index` — [safe], no recreate.
**Tests:** migration applies cleanly; `compare_metadata == []`; (optional) the indexes exist via a
PRAGMA/inspect assertion.
**Commit:** `perf(N2): covering indexes for queue/documents/applications`.

### N2 — explicitly deferred within T2b (too risky / low value for now)
- `String(n)` length caps on free-text columns (truncation risk on existing data).
- `server_default`/`onupdate` on timestamp columns (table-recreate cost; Python defaults already work
  for the app — the only gap is raw-SQL inserts, which we don't do). May revisit if it bites.
- Singleton `CHECK(id = 1)` on `UserProfile`/`SearchSettings` and boot-seeding (ergonomic, not
  integrity-critical).

### N2 verification
`uv run pytest -q` green incl. new constraint tests; `uv run python -m alembic upgrade head` clean on
a fresh DB AND on a copy of an existing DB (the data-cleanup path exercised); `compare_metadata == []`;
pyright at/under ceiling.

## 7. Sequencing & dependencies
N1 first (cheap, isolated). N2 next (self-contained DB work; T2b lots are independent but share the one
migration revision, so run them in order and keep the migration coherent). N3 last (it validates the
finished whole and is the only milestone needing out-of-sandbox actions). Merge `feat/v1.1-finishing-touches`
into `main` locally after N1+N2; N3's push is a separate, user-approved step.

## 8. Success criteria recap
§2's five bullets hold; the branch merges cleanly into `main`; pyright ceiling ratcheted down and
enforced by CI; the schema is constraint-hardened with a clean, idempotent, data-safe migration.
